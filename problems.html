Problems

Third Party Criticism

Find Sneer criticism by other people: http://www.google.com/search?q="Sneer+problems"
We value feedback a lot. If you post criticism about Sneer, do not forget to add the phrase "Sneer problems", as it will help people find it using the link above.
It doesn't even have to be constructive, but at least don't make a fool of yourself:
  • Don't Slander - Give concrete examples of the problems you are talking about. Don't just say "Sneer is full of bugs" or "Sneer sucks" without pointing the bugs or elaborating on your complaints.
  • Know What You Are Talking About - Don't just randomly walk around the code pointing fingers at snippets out of context. We have already seen, for instance, violent criticism of exception handling snippets in code that, when executed, produces the console message "Under Construction. Don't run in production". :-P
  • Be Honest - Don't post known problems - such as the ones below - as if you had discovered a big hidden secret.

Security

Security hooks are built into Sneer from the beginning:
  • Each brick runs in its own classloader so that it can be sandboxed using the Java security model. However, Sneer is NOT sandboxing them yet.
  • All data communication passes through a single point in the code, allowing data encryption to be added when it becomes a priority.
  • Each Sneer user has their own "Seal" (a SHA512 hash of an ECDSA512 public-key) for authentication, although it is NOT being enforced right now.
So, simply DO NOT run 3rd party bricks you don't trust and do not use Sneer to host or transfer security-critical information just yet.

GUI Freeze-ups

The GUI is not stable yet and will occasionally freeze-up. We are pushing the limits of existing GUI frameworks, which are mostly geared toward having the GUI initiate events and not asynchronously reacting to events in the system.

Network Hogging

Sneer is currently opening an inordinate amount of TCP connections and it might hog old, weaker routers.
